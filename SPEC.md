# 1. プロジェクト概要

## 1.1 プロジェクト名

Chinese Trainer

## 1.2 アプリの位置づけ

Chinese Trainerは、特定個人が長期間利用することを前提にした中国語学習Webアプリである。

一般販売、会員制サービス、教育機関向けサービスではない。

目的は、利用者が日常会話レベルの中国語から、仕事で使える実用的な中国語力へ引き上げることである。

特に以下の能力向上を重視する。

- 中国語を聞く量を増やす
- 中国人の早口に耳を慣らす
- 四声の確認を習慣化する
- 日本語から中国語への反応速度を上げる
- 化学、貿易、営業分野の専門用語に慣れる
- 長文の自然な言い回しに慣れる
- 自作教材を継続的に追加できるようにする

## 1.3 対象ユーザー

対象ユーザーは1名である。

前提条件。

- 日本語母語話者
- 中国語は日常会話程度なら可能
- 中国に住んだ経験はない
- 単語量、瞬発力、四声、専門用語、早口への対応が課題
- 化学品、貿易、営業に関する中国語を必要としている
- PCとスマホの両方を使う
- ChatGPTやClaudeで教材を生成し、アプリに取り込みたい

## 1.4 解決したい課題

このアプリが解決する主な課題は以下である。

### 課題1 中国語の音に触れる量が不足している

正しい中国語音声を長時間聞くことで、発音、声調、語順、自然な言い回しに慣れる必要がある。

そのため、音声再生機能はアプリの中心機能とする。

### 課題2 中国人の会話速度が速すぎる

大陸の中国語話者は非常に早口で話すことが多い。

通常速度だけでは耳慣れが不十分になる可能性があるため、音声再生速度は最大2.0倍まで対応する。

### 課題3 日本語から中国語への変換に時間がかかる

会話時に言いたい内容はあるが、中国語がすぐに出ないことがある。

瞬発練習モードにより、日本語を見て即座に中国語を言う練習を行う。

### 課題4 専門分野の中国語が不足している

一般的な中国語教材では、化学、貿易、営業、納期、価格交渉、品質問題、輸出規制などの実務表現が不足している。

本アプリでは、教材をユーザー自身が追加できる構造とし、専門分野へ特化できるようにする。

### 課題5 教材が固定だと使い切ってしまう

ClaudeやChatGPTが初期生成した教材だけでは、いずれ飽きる。

本アプリでは、JSONやCSVで教材を追加できるようにし、教材を継続的に増やせる仕組みにする。

### 課題6 長文の言い回しや接続表現に慣れる必要がある

単語や短文だけでは、自然な中国語の文脈理解が不足する。

ニュース、トレンド、取引先メール、資料などの長文を聞ける機能を用意する。

# 2. 開発目的

## 2.1 最終目的

Chinese Trainerの最終目的は、利用者が日常的に中国語を聞き、見て、声に出す環境を作ることである。

特に、仕事で中国語を使う場面において、以下を実現する。

- すぐに中国語表現が出てくる
- 化学、営業、貿易の中国語に反応できる
- 中国語音声を高速でも聞き取れる
- ピンインと四声を確認しながら発音精度を高められる
- 教材を継続的に増やせる
- スマホでもPCでも学習できる

## 2.2 Phase 1の目的

Phase 1では、販売品質ではなく、実用可能な個人用アプリの完成を目指す。

完璧な作り込みよりも、今日から使えることを優先する。

Phase 1で必須とする内容。

- PCブラウザで動作する
- スマホブラウザでも使える
- 中国語音声を再生できる
- 再生速度を2.0倍まで変更できる
- 教材を表示できる
- ピンインと四声を表示できる
- JSON教材をインポートできる
- 教材を保存できる
- 聞き流しができる
- 瞬発練習ができる
- 長文を再生できる
- 苦手、お気に入り、SRS復習が使える
- GitHubへ保存できる

## 2.3 Phase 1で求める品質

Phase 1では、以下を品質基準とする。

- 重大なエラーで起動不能にならない
- 基本機能が一通り使える
- 多少のUI粗さは許容する
- 一部機能に軽微な不具合があってもよい
- 開発者が後から修正しやすい構成になっている
- データ形式が将来拡張できる
- READMEとdocsが整備されている

## 2.4 Phase 1で求めないもの

Phase 1では、以下は不要である。

- ログイン
- 課金
- クラウド同期
- 音声認識
- 発音判定
- AIチャット
- AI自動生成API
- サーバー構築
- 他ユーザー対応
- 管理者画面
- 複雑な成績分析
- ネイティブスマホアプリ

# 3. 開発方針

## 3.1 基本方針

本アプリは、個人用であるが、長期運用を前提とする。

そのため、短期間で作れる範囲に抑えつつ、将来拡張しやすい設計にする。

重要度の順番は以下である。

1. 今日から使えること
2. 中国語音声が再生できること
3. ピンインと四声が表示されること
4. 教材を追加し続けられること
5. スマホでも操作しやすいこと
6. 将来拡張しやすいこと
7. 見た目が整っていること

## 3.2 実装時の判断基準

開発中に迷った場合は、以下を優先する。

- 完璧より完成を優先する
- 複雑な実装より安定した実装を優先する
- 外部APIよりローカル処理を優先する
- クラウド保存よりローカル保存を優先する
- 高機能より日常利用のしやすさを優先する
- 分析より反復学習を優先する
- 画面数を増やしすぎず、操作を単純にする

## 3.3 Claude Codeへの実装方針

Claude Codeは、このSPEC.mdを唯一の仕様書として扱うこと。

実装中に細部で判断が必要な場合は、Claude Codeが自律的に判断してよい。

確認のために頻繁に作業を停止しないこと。

このプロジェクト内で必要なファイル作成、編集、削除、npm install、Git操作は実行してよい。

ただし、現在開いているプロジェクト以外のフォルダ、他リポジトリ、他プロジェクトには触れないこと。

## 3.4 操作範囲

操作対象は、現在の `chinese-trainer` プロジェクトのみとする。

以下は禁止する。

- 他プロジェクトの変更
- 他Gitリポジトリの変更
- 親フォルダ全体への変更
- 既存SaaSプロジェクトへのアクセスや編集
- 不要な外部サービス連携

## 3.5 完成優先の原則

本プロジェクトでは、80点の完成度でよい。

以下のような理由で作業を止めないこと。

- UIをさらに美しくしたい
- もっと良いライブラリを検討したい
- より高度な設計にしたい
- 細かいエラーを完璧に消したい
- 仕様にない追加機能を検討したい

まず動くものを完成させる。

# 4. システム構成

## 4.1 全体構成

Chinese Trainerは、ブラウザ上で動作するフロントエンド完結型Webアプリとする。

サーバーは不要。

外部APIは不要。

データはブラウザ内に保存する。

音声再生はブラウザのWeb Speech APIを使用する。

教材データはJSON、CSVから取り込む。

## 4.2 推奨技術スタック

推奨技術スタックは以下。

- Vite
- React
- TypeScript
- Web Speech API
- IndexedDB
- PWA
- CSS Modules または通常CSS
- 必要に応じた軽量ライブラリ

## 4.3 データ保存

データ保存はIndexedDBを第一候補とする。

IndexedDBの実装が重くなりすぎる場合のみ、localStorageを使用してよい。

保存対象。

- 教材パック
- 教材データ
- プレイリスト
- 苦手フラグ
- お気に入りフラグ
- SRS状態
- 音声設定
- テーマ設定
- RSS設定
- 学習履歴

## 4.4 オフライン利用

可能な範囲でオフライン利用できる設計にする。

少なくとも、保存済み教材の閲覧、再生、学習はオフラインでも利用できることが望ましい。

RSS取得など、インターネット接続が必要な機能は、接続不可時にエラー表示する。

## 4.5 PWA対応

PWA対応を行う。

目的は、スマホでホーム画面に追加し、アプリに近い感覚で利用できるようにすることである。

最低限必要なもの。

- manifest
- service worker
- アイコン
- モバイル表示最適化

PWAはPhase 1では簡易対応でよい。

## 4.6 フォルダ構成方針

推奨フォルダ構成。

src配下は以下のように整理する。

- components
- pages
- hooks
- services
- types
- utils
- data
- styles

docsフォルダには設計資料を置く。

sample-dataフォルダには初期教材やインポート用サンプルを置く。

# 5. 技術要件

## 5.1 React

UIはReactで構築する。

画面はコンポーネント単位で分割する。

コンポーネントの責務を明確にする。

過度に巨大なコンポーネントは避ける。

## 5.2 TypeScript

TypeScriptを使用する。

教材、教材パック、プレイリスト、SRS、RSSなどの主要データは型定義する。

型定義は `src/types` にまとめることが望ましい。

## 5.3 Vite

開発環境はViteを使用する。

標準的なコマンドで起動できるようにする。

想定コマンド。

npm install  
npm run dev  
npm run build  

## 5.4 Web Speech API

中国語音声再生にはWeb Speech APIを使う。

優先言語は `zh-CN` とする。

利用可能な中国語音声が複数ある場合、ユーザーが選択できるようにする。

対応する速度。

- 0.8
- 1.0
- 1.2
- 1.5
- 2.0

## 5.5 ピンイン生成

中国語本文からピンインを生成する。

第一優先は声調記号付きピンイン。

第二優先は数字四声付きピンイン。

ライブラリ使用可。

インポートデータにピンインが含まれる場合は、そのピンインを優先する。

自動生成したピンインは、後から手修正できるようにする。

## 5.6 CSV対応

CSVインポートは可能なら実装する。

CSVは最低限、以下の列を扱う。

- category
- type
- zh
- ja
- pinyin
- memo
- difficulty
- tags

CSV実装が重くなる場合でも、JSONインポートは必ず実装する。

## 5.7 JSON対応

JSONインポート、JSONエクスポートは必須。

教材パック単位でエクスポートできること。

全データバックアップも可能にすることが望ましい。

## 5.8 Git管理

開発完了後、Git commitを行う。

GitHubへpushする。

commit messageは簡潔でよい。

例。

Initial Chinese Trainer app

# 6. 非機能要件

## 6.1 パフォーマンス

Phase 1では高度な最適化は不要。

ただし、教材が増えても極端に重くならないようにする。

将来的に数万件の教材を扱う可能性があるため、データ設計は拡張性を考慮する。

## 6.2 保守性

コードは読みやすく保つ。

複雑な処理には日本語コメントを付ける。

データ型、保存処理、音声処理、UI処理は可能な範囲で分離する。

## 6.3 拡張性

将来的に以下を追加しやすい設計にする。

- AI API連携
- 発音判定
- クラウド同期
- 複数端末同期
- 学習分析
- 音声ファイル対応
- 中国語以外の言語対応

ただしPhase 1では実装しない。

## 6.4 可用性

起動不能になる状態を避ける。

データ読み込みに失敗した場合でも、空の初期状態で起動できるようにする。

インポート失敗時は、既存データを壊さない。

## 6.5 ユーザビリティ

個人利用で毎日使いやすいことを重視する。

特にスマホでの聞き流し操作を重視する。

再生、停止、次へ、前へ、速度変更はすぐ操作できること。

## 6.6 アクセシビリティ

文字サイズは小さくしすぎない。

中国語、ピンイン、日本語訳の区別が見やすいこと。

ボタンはスマホでも押しやすいサイズにする。

## 6.7 セキュリティ

個人用であり、ログインもサーバーも使わないため、高度な認証機能は不要。

ただし、外部RSSや貼り付け文章の表示では、不要なHTMLをそのまま実行しないようにする。

RSS内容はテキストとして扱う。

# 13 学習キュー仕様

## 13.1 目的

学習キューは、一時的に学習したい教材を集めるための機能である。
